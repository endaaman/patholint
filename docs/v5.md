
# 病理診断報告書コンプライアンス検証システム研究メモ

**タイトル** Development of a Local LLM-based Validation System for Pathology Report Compliance

---

## 1. 背景・問題提起

### 1.1 病理レポートエラーの頻度と臨床的インパクト

病理報告書の品質問題は、頻度・インパクトともに無視できない規模で存在する。

**頻度：**

- CAP Q-Probes研究（73施設・36万件）では、レポート欠陥率は1000件あたり4.7件（中央値5.7）と報告されている（Volmar et al. _Arch Pathol Lab Med._ 2014）。欠陥の内訳は「記載上の問題」が58.4%と最大であり、誤診断（14.6%）や検体誤識別（13.3%）を上回る。すなわちエラーの過半数は**記載・表現レベルの問題**であり、LLMによる自動検証が最も効果を発揮しうるカテゴリである。
- 病理全体のエラー率は約5%と推計され、後分析フェーズ（報告）だけで2.0%に相当する（Alaghehbandan & Raab SS. _PSNet/AHRQ._ 2013）。同報告では「測定プロセスが統一されていないため頻度と影響が十分に把握されていない」とも明記されており、**定量化されていないこと自体が問題の放置につながっている**という論法が成立する。

**臨床的インパクト：**

- MD Anderson Cancer Centerでの前向き研究（2718件）では、25%に何らかの不一致が認められ、うち6.2%（169件）が治療方針を変える重大な診断変更につながった（Middleton et al. _J Oncol Pract._ 2014）。
- 外部QAプログラムの系統的データでは、major discordance率は臓器群により3〜9%の範囲であり、消化管は最高率群に含まれる（Peck et al. _J Clin Pathol._ 2018）。

→ **記載レベルの問題が最多であり、かつ臨床的に重大な帰結をもたらしうる。**  
→ 自動検証システムによる介入の必要性はデータで裏付けられる。

### 1.2 臨床的背景（大腸癌・日本の状況）

- 外科医（特に消化器外科）は病理医に対して「**規約に沿った診断**」を明示的に要求する
    
- 病理診断報告書には、取扱い規約に基づく**診断サマリー行**の正確な記載が必須
    
- 例（大腸癌取扱い規約第9版）：
    
    ```
    腹腔鏡下右結腸切除術, A, 2型, 47×30 mm, 85%, tub2>tub1,pT3, INFb, Ly1a, V1a, Pn1a, PM0(125mm), DM0(75mm), RM0(3mm),pN0(0/9), cM0, pStageIIA, R0.
    ```
    
- この診断サマリー行が**正しく・網羅的に記載されること**が最小条件
    
- 国内では病理レポートのテキスト品質管理に関する研究はほぼ未着手であり（日本病理学会. 病理診断支援AIの手引き. 2021）、JP-AID事業の対象は画像AI診断精度の管理が中心で、レポートテキストの記載不備・矛盾検出は射程外である
    

### 1.3 報告書の構成と要求水準

|構成要素|要求水準|備考|
|---|---|---|
|診断サマリー行|**必須**。全規約項目の存在と整合性|常に記載される|
|肉眼所見|矛盾がないこと|システムにより記載される場合とされない場合がある|
|組織所見|矛盾がないこと|省略も許容。外科医はあまり参照しないが、書かれている以上は整合性が求められる|

### 1.4 既存手段の限界

**シノプティックレポート（チェックリスト方式）：**

- 必須項目の記載完全率は有意に改善する（van Leersum et al. _Virchows Arch._ 2016; 33研究の系統的レビュー）
- テンプレート導入で必須項目完全率が77%→98%に改善した報告もある（Hofman et al. _Mod Pathol._ 2024）
- しかし「ほとんどの研究で90%を超えられない」という指摘もあり（Srigley et al. _JCO Clin Cancer Inform._ 2018）、否定語（no/not）の扱いや同一データの重複記載が矛盾エラーの源となることも示されている
- **決定的な限界：** 既存研究の評価対象はすべて「記載項目の有無」であり、**自由記載内の矛盾・不整合の検出は評価外**（van Leersum et al. 2016）

→ シノプティックレポートはDeficiencyを減らすが、InconsistencyとRuleViolationには対応できない  
→ **自由記載間の論理的整合性を検証できる仕組みが未解決として残されている**

### 1.5 「修正」は別次元である

- 矛盾が発見された場合、どちらの記載が正しいかは**外部情報（実際の標本・所見）**に依存する
- LLMが「治す」ことはできない（すべきでもない）
- システムの責務は **Validation（検証・問題提示）に限定**することが設計として誠実
- UXとしては、YES/NO・選択肢提示によるフィードバックが現実的
- 修正機能はアプリケーション実装側の設計問題として切り離す

## 2. 問題の形式化（AI以前の定義）

AI実装の前に、**「何が問題か」を操作的に定義すること自体が研究貢献**となる。

### 2.1 Invalidityの定義と4分類（確定版）

本システムが検出する問題の総称を **Invalidity** とする。Invalidityとは、**報告書テキストとルールセットの整合性、および報告書内部の整合性（integrity）が損なわれた状態**を指す。

> **Validationのスコープ：**  
> 本システムが保証するのは、**報告書テキスト内部の記載、およびルールセットとの整合性（integrity）**に限定される。検体・患者の同一性、診断の臨床的正否、所見の真偽など、外部情報に依存する判断はすべてスコープ外である。これはシステムの限界ではなく、**Validationの責務を誠実に限定する設計判断**である。

4分類は**完全に排他**である。

|分類|定義|検出に必要なもの|例|
|---|---|---|---|
|**RuleViolation**|該当箇所の記載が存在し、ルールセットに照らして値・表現・形式が不正|ルールセット参照|pT3なのに「pT1」と記載、Stage計算の誤り|
|**Typo**|スペルミス・転記ミスなど、ルールセットを参照せずとも記載自体から判定可能な表記上の誤り|記載自体|Adenocaricnoma のスペルミス、数値の転記ミス|
|**Deficiency**|必須項目に対応する記載がレポート内に存在しない|ルールセット参照|cM項目の記載がない|
|**Inconsistency**|個々の記載はルールセット上合法だが、複数箇所を突き合わせると矛盾が生じる|レポート内部の関係推論|Ly1aと記載したが組織所見に脈管侵襲の記載なし|

**「記載がない」の操作的定義：**  
その項目に言及する文字列がレポート内に存在しない。「所見として認めない」旨の陰性記載はDeficiencyではなく合法な記載として扱う。自由記載形式において「空欄フィールド」という状態はそもそも発生しない。

**実験仮説との対応（各分類が1対1で対応）：**

|分類|予測される振る舞い|
|---|---|
|**Typo**|どのモデルでも検出可能。条件差が小さい。モデルの日本語基礎能力のベースライン指標となる|
|**RuleViolation**|規約情報ありで検出率が上昇する（ルールセット依存）|
|**Deficiency**|規約情報ありで検出率が上昇する（ルールセット依存）|
|**Inconsistency**|モデル規模が大きいほど検出精度が向上する（推論力依存）|

### 2.2 既存taxonomyとの差別化

Meier et al.（_Am J Clin Pathol._ 2008）のtaxonomyは病理レポート修正報告（amended reports）を対象とした帰責・原因フェーズベースの事後QA監査向け分類（Misinterpretation / Misidentification / Specimen defect / Other report defects）であり、リアルタイムな自動検証システムの出力形式には対応していない。

本研究の4分類（RuleViolation / Typo / Deficiency / Inconsistency）は **LLMのサジェスト出力形式に対応した検出ベースの操作的定義** であり、目的が根本的に異なる。上位概念としてInvalidityを用いることで、先行研究のDefect・Errorといった語彙との混同も回避している。

→ 「既存taxonomyはリアルタイム自動検証システムの出力形式に対応していない」というギャップとして論文に記述できる。

### 2.3 スコープの明示

**対象（In scope）：報告書テキストのintegrity**

|integrity の種類|内容|
|---|---|
|ルールセットとの整合性|RuleViolation・Deficiencyの検出対象|
|レポート内部の整合性|Inconsistencyの検出対象|
|記載自体の正確性|Typoの検出対象|

**対象外（Out of scope）：外部情報に依存するすべての判断**

|項目|理由|
|---|---|
|検体・患者の同一性|外部情報（物理的検体・台帳）に依存|
|診断の臨床的正否|外部情報（標本観察・臨床情報）に依存|
|どちらの記載が正しいか|外部情報（実際の所見）に依存|
|重大度の判定|臨床文脈・施設慣習・患者背景に依存し、レポートテキスト単独からの決定は原理的に不可能|
|「治す」こと|外部情報と設計判断に依存|

### 2.4 ルールセットの扱い

- ルールセットはad hocに差し替え可能（規約・ハウスルールを**同一の仕組みで扱う**）
- RuleViolation・Deficiencyは常に「どのルールセットに対して」という相対的な判定
- **ハウスルール**（記載順序・主治医通知ルールなど）を今回のスコープに含めるかは引き続き検討事項

## 3. 技術設計

### 3.1 コンテキスト設計の基本方針

LLMに何を渡すか：

| 入力         | 扱い                         |
| ---------- | -------------------------- |
| 報告書本文      | **常に渡す**（コンテキストから漏れない）     |
| 規約（生の全文）   | 使わない（数百ページ・コンテキスト圧迫・コスト増大） |
| 圧縮済みルールセット | **実験条件として使用**（あり/なしで比較）    |
| ハウスルール     | 同一の仕組みで追加可能（今回のスコープ検討中）    |

### 3.2 規約の圧縮・形式化

生の規約全文 → 手動でミニマムルールセットに圧縮する作業が必要。  
この圧縮作業自体が「**規約の操作的定義**」という研究貢献として位置づけられる。

```
例：大腸癌取扱い規約 圧縮ルールセット（イメージ）
- 必須記載項目: pT, pN, cM, Ly, V, Pn, INF, RM, PM, DM, Stage, R分類
- pTの定義:
    T1a: 粘膜下層1/3以内
    T1b: 粘膜下層1/3超
    T3: 固有筋層を超えて浸潤
    ...
- Stageは pT × pN × cM から規約表に従い自動決定
- Ly/V/Pnはそれぞれ 0/1a/1b/2 で記載
- 断端: PM/DM/RMそれぞれ距離をmmで記載（0の場合は浸潤なし）
```

### 3.3 規約の選択・管理

- 臓器別・術式別に圧縮済みルールセットをあらかじめ用意
- どの規約セットを使うかは**手動選択でよい**（病理では臓器が自明）
- 自動ルーティング（報告書を見て規約を自動選択）は将来課題

### 3.4 出力形式

LLMの出力は**行ベースのタグ付きテキスト**を要求する。

```
[RuleViolation] 具体的な問題の説明
[Typo] 具体的な問題の説明
[Inconsistency] 具体的な問題の説明
[Deficiency] 具体的な問題の説明
```

**設計根拠：**

- 行単位で独立してパース可能（1行が壊れても他の行は救える。JSONだと全滅する）
- パースが単純（正規表現 `^\[(RuleViolation|Typo|Inconsistency|Deficiency)\] (.+)$` で処理可能）
- ローカルモデルとの相性が良い（括弧の対応やカンマ管理が不要で自然言語に近い）
- 人手での採点にも支障がない

**問題がない場合の出力：** `問題なし` とだけ出力させる。

## 4. 実験設計

### 4.1 実験条件（2×3 マトリクス）

| ローカル（民生GPU / 20B）     | オンライン（GPT-4 / Claude） | H200×8（70B+） |      |
| --------------------- | --------------------- | ------------ | ---- |
| **設定1: 規約情報なし**       | 条件A1                  | 条件B1         | 条件C1 |
| **設定2: 規約情報あり（圧縮済み）** | 条件A2                  | 条件B2         | 条件C2 |

**設定1（Zero-shot）の操作的定義：**

- プロンプトには「この病理報告書に問題点があれば、1行に1つずつ記載してください。問題がなければ『問題なし』とだけ出力してください。」とのみ記載
- 分類タグ（RuleViolation / Typo / Inconsistency / Deficiency）は**指示しない**。モデルが分類名を付けた場合はそれを記録するが、付けなかった場合は人手で分類する
- ルールセットは渡さない
- モデルが元来持っている医学知識のみで検出する条件

**設定2（ルールセット付き）の操作的定義：**

- プロンプトに圧縮済みルールセットを含め、「以下のルールに照らして問題点を指摘してください」と指示
- 出力形式として `[RuleViolation]` `[Typo]` `[Inconsistency]` `[Deficiency]` のタグを指定
- 問題がない場合は「問題なし」とだけ出力させる

### 4.2 評価指標

#### 主要指標：Invalidityの検出性能

各条件において、RuleViolation / Typo / Inconsistency / Deficiency ごとに算出する。

|指標|定義|
|---|---|
|**感度 (Recall)**|TP / (TP + FN)。Gold Standardに存在するInvalidityのうち、正しく検出された割合|
|**FP率**|FP件数 / 症例数。1症例あたりの誤検出数。特異度の代替として使用する（※）|

> **※ 特異度を使わない理由：** 特異度の分母は「真に問題のない項目の総数（TN + FP）」だが、自由記載レポートにおいて「問題がありうる箇所の全数」を客観的に定義することが困難である。そのためTNが定まらず、特異度は計算不能。代わりにFP率（症例あたりFP件数）を報告する。

#### FPの下位分類

|分類|定義|例|
|---|---|---|
|**FP-Relevant**|医学的に妥当だが、Gold Standardのルールセット（大腸癌取扱い規約第9版）の範囲外の指摘|WHO分類の組織型記載がない、CRM距離が未記載|
|**FP-Spurious**|事実と異なる、医学的に誤っている、または意味不明な指摘|Ra＋低位前方切除が不整合と誤認|

> **FP-Relevantの解釈：** Zero-shot条件ではFP-Relevantが多数出ることが予想される。これは「モデルが独自の基準で判定している」ことを示し、「ルールセットを与えることでFPが減少する」という設定1→設定2の効果として解釈できる。FP-Relevantの大半は、ルールセットのスコープを広げれば吸収されるものであり、モデルの能力の問題ではなくスコープの問題である。

#### 副次指標：パースエラー率

|指標|定義|
|---|---|
|**パースエラー率**|パース失敗行数 / (パース成功行数 + パース失敗行数)|

- **パース成功：** 行頭に `[RuleViolation]` `[Typo]` `[Inconsistency]` `[Deficiency]` のいずれかがある行
- **パース失敗：** 上記タグがないが、何らかの指摘内容を含む行（人手で判定）
- **除外：** 指摘を含まない応答（要約・挨拶など）、空行、「問題なし」応答はパースエラーに含めない

> **設定1ではタグ指示をしないため、パースエラー率は設定2のみで算出する。** 設定1の出力はすべて人手で採点・分類する。

### 4.3 実験仮説

|比較軸|仮説|対応する指標|
|---|---|---|
|設定1 → 設定2|RuleViolation・Deficiencyの感度が上昇する|感度の条件間差|
|設定1 → 設定2|FP（特にFP-Relevant）が減少する|FP率の条件間差|
|設定1 → 設定2|Typoの感度は変化しない（規約情報に依存しないため）|感度の条件間差（対照群）|
|モデル規模 小→大|Inconsistencyの感度が向上する（推論力依存）|感度のモデル間差|
|モデル規模 小→大|Typoの感度がベースラインとして機能する（日本語基礎能力の指標）|感度のモデル間差|
|モデル規模 小→大|パースエラー率が低下する|パースエラー率のモデル間差|
|**どちらの効果が大きいか**|← メインメッセージ候補||

### 4.4 データセット設計

#### 概要

大腸癌取扱い規約第9版に準拠した仮想症例データセットを作成する。実際の大腸癌病理報告書をベースとし、正常レポートとエラー埋め込みレポートのペアで構成する。

#### 症例構成

|カテゴリ|症例数|目的|
|---|---|---|
|正常レポート（エラーなし）|10例|FP率の評価（特異度の代替）|
|エラー埋め込みレポート|20例|感度の評価|
|**合計**|**30例**||

#### エラー埋め込みの設計

- 1症例あたり **1〜3個** のInvalidityを埋め込む（複数カテゴリの混在あり）
- 3分類それぞれが十分な検出機会を持つようバランスを取る

|分類|埋め込み総数（目安）|例|
|---|---|---|
|RuleViolation|8〜10個|pT値の不正、Stage計算の誤り、Ly/V/Pnの不正値|
|Typo|4〜6個|スペルミス、数値の転記ミス|
|Deficiency|8〜10個|cM欠落、R分類欠落、INF欠落|
|Inconsistency|8〜10個|pT2だが「漿膜に達する」、Ly1aだが「脈管侵襲なし」|

#### ベースレポートの作成方法

1. 実際の自分の大腸癌レポートを匿名化して収集（10〜15件）
2. 各レポートについて、エラーを除去した正常版を作成
3. 正常版に意図的にInvalidityを埋め込んだエラー版を作成
4. 同一ベースから正常版・エラー版のペアが生まれる（レポートの使い回し効率が良い）

> **自然発生エラーの扱い：** 実際のレポートに存在するエラー（cM欠落、スペルミス等）が発見された場合、そのレポートをエラーあり症例としてGold Standardに含める（人工的エラーと自然発生エラーを区別して記録する）。

#### Gold Standardラベルの形式

各エラー埋め込みレポートに対し、以下の形式でラベルを付与する：

```yaml
case_id: "CRC-001"
base_report: "正常版CRC-001"
invalidities:
  - type: "RuleViolation"
    location: "診断サマリー行"
    detail: "pStage ⅢA → 正しくは pStage ⅢB（pT3N1aM0の場合）"
  - type: "Deficiency"
    location: "診断サマリー行"
    detail: "cM の記載がない"
```

## 5. システム設計・実装提案

### 5.1 アーキテクチャ概要

```
[報告書入力]
     ↓
[臓器・規約の選択] ← 手動 or 将来的に自動ルーティング
     ↓
[圧縮済みルールセット + 報告書 → LLM]
     ↓
[問題点の出力（行ベースタグ付きテキスト）]
  - [RuleViolation] ○○の記載が規約違反
  - [Typo]          △△のスペルミス
  - [Inconsistency] □□と☆☆が矛盾
  - [Deficiency]    ××項目が欠落
     ↓
[採点（人手 or 自動パース → Gold Standardとの照合）]
```

### 5.2 施設規模別モデル選定ガイドライン（仮）

|施設環境|推奨モデル規模|特性|
|---|---|---|
|民生GPU（RTX 4090等）|〜20B|院内完結・患者データ外部送信なし・低コスト|
|オンラインAPI|GPT-4 / Claude|高精度・従量コスト・プライバシー要件の検討必要|
|研究用GPU（H200×8等）|70B+|最高精度・比較ベンチマーク用|

### 5.3 プライバシーの考慮

- ローカルLLMは患者データの外部送信を回避できる
- 守られる情報の範囲を明確化する必要がある（氏名・診断名・施設情報など）
- オンラインモデル使用時のde-identification要件を別途整理

## 7. 発表構成（スライド案）

1. **背景** — 病理レポートエラーの頻度・インパクト（①②③④）、外科医の要求
2. **既存手段の限界** — シノプティックでは内的矛盾を検出できない（⑤⑥⑦）
3. **問題の形式化** — Invalidity（RuleViolation / Typo / Deficiency / Inconsistency）の4分類、スコープ・integrityの明示（⑨と差別化）
4. **コンテキスト設計** — LLMに何を渡すか、規約圧縮の必要性と貢献
5. **実験設計** — 2×3条件、出力形式、評価指標
6. **結果** — 条件別スコア比較（分類×モデル×規約あり/なし）、FP内訳（Relevant vs Spurious）、パースエラー率
7. **考察** — 規約情報 vs モデル規模の効果比較、Inconsistencyの特殊性、FP-Relevantの解釈
8. **設計提案** — Validationに責務を限定する思想、UX、施設規模別ガイドライン
9. **結論・展望** — 他臓器汎化、ハウスルール対応、自動ルーティング
